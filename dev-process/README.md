# dev-process — AI 駆動開発ハーネス（最小構成）利用手順

Fable 5 で P0〜P13 を実装したときの開発プロセスを、**他の LLM モデルでも同等の品質で
再現する**ための最小ハーネスです。大掛かりな仕組みは使わず、次の 4 ファイルだけで動きます。

| ファイル                      | 役割                                                                 |
| ----------------------------- | -------------------------------------------------------------------- |
| `dev-process/PROCESS.md`      | 開発サイクル 7 ステップ + 品質基準 + 禁止事項（**唯一の正**）        |
| `dev-process/STATE.md`        | 申し送り（進捗・恒久制約・ハマりどころ・残課題）。フェーズごとに更新 |
| `dev-process/verify.mjs`      | 品質ゲートの 1 コマンド化（静的チェック→ユニット→ビルド→E2E）        |
| `AGENTS.md`（リポジトリ直下） | 各種エージェントが自動で読む入口。上記 2 つへ誘導                    |

Claude Code 用に `.claude/skills/dev-cycle/SKILL.md`（`/dev-cycle`）も同梱しています。

## 仕組みの考え方

品質を担保しているのは高度な仕掛けではなく、次の 3 点です。モデルが変わっても
この 3 点を強制できれば品質は近づきます。

1. **順序の強制** — 要件確認（SRS ID の原文確認）→ 既存パターン調査 → 実装、の順を守らせる。
   PROCESS.md の各ステップに「完了条件」を付け、飛ばし読みを防ぐ。
2. **機械的な合否判定** — 「動いたはず」を禁止し、`verify.mjs --e2e` の ALL PASS だけを
   完了条件にする。ABI 切替のような踏み外しやすい手順もスクリプトに封じ込める。
3. **申し送りの外部化** — Fable 5 のセッション記憶に相当する情報を STATE.md として
   リポジトリに置き、毎フェーズ更新をプロセスに組み込む（更新忘れ = Done にならない）。

## 利用手順

### 共通（どのモデルでも）

1. 新しいセッションを開始したら、まず `dev-process/STATE.md` を読ませる。
2. 指示は「**PROCESS.md に従って P14 を進めて**」のように、対象フェーズ + PROCESS.md 準拠を明示する。
3. 完了報告を受けたら、次の 3 点を確認する（レビュー観点）:
   - `node dev-process/verify.mjs --e2e` が ALL PASS か（自分でも再実行できる）
   - 報告に「見送った項目（残課題）」が明記されているか
   - `dev-process/STATE.md` が更新されているか

### Claude Code（Fable / Opus / Sonnet / Haiku）

- `/dev-cycle P14を進めて` と打つ（または「P14を進めて」だけでもスキルが起動する）。
- モデルを切り替えても手順は同じ。能力の低いモデルほど、1 回の指示を
  フェーズ丸ごとではなく**タスク単位**（例: `/dev-cycle P14-1 のみ`）に絞ると品質が安定する。

### Codex CLI / Cursor / GitHub Copilot / Gemini CLI など

- これらのツールはリポジトリ直下の `AGENTS.md`（Cursor は Rules 経由でも可）を自動で読む。
  追加設定なしで PROCESS.md / STATE.md へ誘導される。
- 読み込まれない環境では、最初のプロンプトに次を貼る:
  > まず dev-process/STATE.md と dev-process/PROCESS.md を読み、PROCESS.md の
  > 7 ステップに従って <対象フェーズ> を実装してください。完了条件は
  > `node dev-process/verify.mjs --e2e` の ALL PASS と STATE.md の更新です。

### チャット UI（ツール実行なし）のモデルに手伝わせる場合

コードを直接実行できないモデルには、ステップ 1〜3（要件整理・設計・実装案）まで任せ、
ステップ 4〜7 は人間が `node dev-process/verify.mjs --e2e` を実行して合否を貼り戻す。

## 品質ゲートの実行

```
node dev-process/verify.mjs         # 静的チェック + ユニットテスト（数分）
node dev-process/verify.mjs --e2e   # 上記 + ビルド + E2E（実アプリ、+2〜3 分）
```

- 途中で失敗すると即打ち切り、末尾に PASS/FAIL 一覧を表示、exit code 1。
- `--e2e` は better-sqlite3 の ABI を Electron へ切替え、終了時（失敗時も）Node へ戻す。

## メンテナンス規約

- プロセスを変えたいときは **PROCESS.md だけ**を編集する（入口ファイルには手順を書かない）。
- 新しい制約・ハマりどころは見つけ次第 STATE.md へ追記する。ここが痩せると
  他モデルでの再現品質が最初に落ちる。
- STATE.md の「フェーズ履歴」は 1 フェーズ 1 行を維持（詳細は git log とコミットメッセージが正）。
