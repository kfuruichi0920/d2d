# 設計情報デジタル化・トレーサビリティ支援ツール 要求仕様書

## 1. 目的

本ツールは、既存のWord、Excel、PowerPoint、Visio、PDF、テキスト等で作成された自然言語中心の設計文書を段階的にデジタル化し、設計要素および設計要素間の関係性を構造化管理することを目的とする。

これにより、要求、制約、機能、構造、振舞、状態、インタフェース、データモデル、検証情報、設計判断根拠等の間に複数観点のトレーサビリティを確保し、仕様変更時の影響分析、設計根拠確認、検証カバレッジ確認、設計レビュー支援を機械的に実施可能にする。

本ツールは、すべてをLLMに自動判断させるものではなく、LLMを候補生成・要約・抽出・レビュー補助に用い、人間による逐次確認、修正、採用、棄却を前提とした human-in-the-loop 型の設計支援ツールとする。

## 2. 基本方針

### 2.1 データ階層

本ツールは、以下の4階層でデータを管理する。

| 階層 | 名称    | 内容                       | 主な役割            |
| -- | ----- | ------------------------ | --------------- |
| ①  | 原本データ | Word、Excel、PDF等の既存文書     | 改変しない一次情報       |
| ②  | 抽出データ | タイトル、本文、図、表、数式、章節構造等     | 原本の外形情報を保持      |
| ③  | 中間データ | チャンク、図表説明、参照関係、段落関係等     | LLM入力および人間レビュー用 |
| ④  | 設計モデル | 要求、制約、機能、構造、振舞、状態、IF、検証等 | トレース・影響分析の正本    |

### 2.2 データ管理原則

1. ①原本データは改変しない。
2. ②抽出データは原本忠実性を重視し、意味解釈を過度に入れない。
3. ③中間データはLLM処理と人間レビューのために整理する。
4. ④設計モデルで設計上の意味を確定する。
5. ④の各要素・関係は、①〜③の根拠情報へ逆参照可能でなければならない。
6. LLMの出力は確定情報ではなく、候補情報として扱う。
7. 人間レビューにより採用・修正・棄却された情報のみを確定設計モデルへ反映する。
8. SQLiteで操作性を確保し、DB to TextによりGit差分確認可能な形式へ出力する。

## 4. システム構成要求

### 4.1 アーキテクチャ方針

本ツールは、マイクロアーキテクチャ方式とし、各機能を個別に有効化・無効化できること。

推奨構成は以下とする。

```text
Desktop Shell
  - Electron または Tauri

Frontend
  - React
  - TypeScript
  - Monaco Editor
  - グリッド表示
  - グラフ表示
  - マトリクス表示
  - Diff表示

Backend / Workers
  - Node.js worker
  - Python worker
  - 必要に応じて Rust / Go worker

Storage
  - SQLite
  - ZIP package
  - JSON / JSONL
  - Markdown / TSV export
  - Git連携用 text dump

LLM Gateway
  - OpenAI
  - Gemini
  - Ollama
  - Azure OpenAI
  - その他社内LLM API
```

## 21. 初期MVPの成功条件

```text
原本文書取込
→ 抽出データ生成
→ 中間データ生成
→ チャンク化
→ LLM候補生成
→ 人間レビュー
→ 設計モデル確定
→ トレース関係編集
→ 欠落分析
→ DB to Text出力
→ Diff確認
→ レポート出力
```

特に、以下を満たすことが重要である。

| 成功条件   | 内容                         |
| ------ | -------------------------- |
| SC-001 | 原本から設計要素まで根拠を追跡できる         |
| SC-002 | LLM候補を人間が採用・修正・棄却できる       |
| SC-003 | 要求から機能、構造、検証までトレースできる      |
| SC-004 | 変更前後の設計モデル差分を確認できる         |
| SC-005 | 変更影響候補を探索できる               |
| SC-006 | Gitでレビュー可能なtext dumpを生成できる |
| SC-007 | 外部LLM API利用有無を制御できる        |
| SC-008 | プラグイン単位で機能を有効／無効化できる       |

## 22. 除外・後回し項目

初期段階では、以下は本格実装の対象外または後回しとする。

| 項目             | 理由                         |
| -------------- | -------------------------- |
| SysMLv2本格編集    | 難易度が高く、内部モデル確立後が適切         |
| 汎用図形編集         | 実装負荷が高く、まずは構造図・状態遷移図に限定すべき |
| Excel完全互換表編集   | 実装負荷が高く、構造化表編集を優先すべき       |
| Word/PDF完全再現出力 | レポート出力から段階導入すべき            |
| 完全自動設計抽出       | LLM誤りが避けられないため、人間レビュー前提とする |
| 高度な権限管理        | 初期は個人または限定チーム利用を想定する       |

# 23. 最終要求サマリ

本ツールは、以下を満たすこと。

1. 既存設計文書を①原本、②抽出、③中間、④設計モデルへ段階的に変換できること。
2. Word、Excel、PDF、PowerPoint、Visio、Markdown、CSV、JSON、画像を扱えること。
3. ①〜③はZIPパッケージで管理し、Git対象外にできること。
4. ④設計モデルはSQLiteで管理し、DB to TextによりGit差分確認可能とすること。
5. LLMは候補生成・要約・抽出・レビュー補助に利用し、確定判断は人間レビューを前提とすること。
6. 要求、制約、機能、構造、振舞、状態、IF、データモデル、検証、用語、設計判断、未決、リスクを管理できること。
7. 設計要素間の関係を明示的に管理し、トレースマトリクス、関係グラフ、階層リスト、CLIクエリで探索できること。
8. 仕様変更時に上流・下流の変更影響分析を支援できること。
9. 原本根拠、LLMログ、レビュー履歴、採否履歴を保持できること。
10. マイクロアーキテクチャとし、各機能をプラグイン単位で有効／無効化できること。
11. デスクトップアプリとして動作し、Web技術によるUI、ローカルSQLite、ローカルワーカー、外部LLM API、ローカルLLMを統合できること。
12. 商用利用可能なライブラリ構成とし、依存ライセンスを管理できること。

以上を本ツールの要求仕様とする。
