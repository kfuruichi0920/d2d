# Word文書情報抽出プラグイン 詳細要求

## 1. 位置づけ

本ドキュメントは、D2D(設計情報デジタル化・トレーサビリティ支援ツール)における Word 文書情報抽出機能の詳細要求を定義する。

Word 文書抽出機能は、`docs/srs_main.md` の以下に対応する詳細要求である。

- `3.1 対象文書` の Word 文書
- `7.1 原本取込` の Word 取込要求
- `7.2 文書構造抽出` の章、節、段落、箇条書き、表、図、図表番号、数式、原本位置抽出要求
- `7.3 抽出結果レビュー` の原本表示と抽出結果表示の並列確認要求
- `8. 中間データ処理要求` のチャンク化、図表説明、参照リンク生成要求

本機能は単独の Flask Web アプリや単独 CLI ツールではなく、D2D のプラグイン構成、ジョブ管理、ZIP パッケージ管理、SQLite 管理、LLM Gateway、レビュー UI と連携する Word 取込・抽出プラグインとして実装する。

## 2. 基本方針

| ID | 要求 |
| --- | --- |
| WDEX-001 | Word 抽出機能は `plugin.word-importer` または同等の Word 取込プラグインとして提供すること。 |
| WDEX-002 | 抽出処理は UI スレッドから分離し、D2D のジョブ管理機構から実行できること。 |
| WDEX-003 | 原本 `.docx` は改変せず、原本ハッシュ、ファイル名、取込日時、抽出器バージョンを保持すること。 |
| WDEX-004 | 抽出結果は、原本忠実な抽出データと、レビュー・LLM 入力向けに整理した中間データを分離して保持すること。 |
| WDEX-005 | 意味解釈、要約、要求候補化は Word 抽出器の責務ではなく、中間データ処理または LLM 支援機能の責務とすること。 |
| WDEX-006 | LLM を利用する場合でも、Word 内部構造の抽出、ID 付与、位置情報保持、表構造復元、参照解決はルールベースで実施すること。 |
| WDEX-007 | LLM 出力は候補として保持し、抽出データそのものを上書きしないこと。 |
| WDEX-008 | 抽出結果には、後続の設計モデル要素が根拠リンクとして参照できる安定 ID を付与すること。 |

## 3. 対象ファイル

| ID | 要求 |
| --- | --- |
| WDEX-FILE-001 | `.docx` 形式の Word 文書を対象とすること。 |
| WDEX-FILE-002 | `.doc` 形式は対象外とし、必要な場合は外部変換器で `.docx` に変換した後に取り込むこと。 |
| WDEX-FILE-003 | パスワード保護、暗号化、破損、OpenXML として不正なファイルは抽出失敗として扱い、ジョブログに理由を記録すること。 |
| WDEX-FILE-004 | ファイルサイズ、ページ数、段落数、表数、画像数など、処理負荷の事前見積りに必要な情報をプリチェックで取得できること。 |
| WDEX-FILE-005 | 巨大文書は一括処理だけでなく、章節または内部 XML パート単位に分割して処理できる設計とすること。 |

## 4. 抽出データモデル

Word 抽出結果は、文書単位のメタデータ、本文要素、注釈情報、メディア情報、参照関係、抽出ログに分けて保持する。

| ID | 要求 |
| --- | --- |
| WDEX-MODEL-001 | 抽出データは JSON / JSONL として保存できること。 |
| WDEX-MODEL-002 | 抽出データは ZIP パッケージ内に格納できること。 |
| WDEX-MODEL-003 | 抽出データの schema_version を保持すること。 |
| WDEX-MODEL-004 | 文書、章節、段落、リスト項目、表、表行、表セル、図、画像、脚注、文末脚注、コメント、変更履歴、ブックマーク、ハイパーリンクに安定 ID を付与すること。 |
| WDEX-MODEL-005 | 各要素は `source_document_id`、`source_path`、`source_hash`、`source_locator`、`extraction_status` を保持すること。 |
| WDEX-MODEL-006 | `source_locator` は、XML パート名、要素インデックス、段落 ID、ブックマーク名、ページ推定値など、原本位置を再確認するための情報を保持すること。 |
| WDEX-MODEL-007 | 抽出時に確定できない情報は空文字で埋めず、未取得、対象外、推定、抽出失敗を区別できる状態値で表すこと。 |
| WDEX-MODEL-008 | Markdown 出力は閲覧・LLM 入力用の派生成果物とし、正本は構造化 JSON / JSONL とすること。 |

抽出データの概念構造は以下を基本とする。

```json
{
  "document": {
    "document_id": "doc-...",
    "file_name": "sample.docx",
    "source_hash": "sha256:...",
    "metadata": {},
    "statistics": {}
  },
  "elements": [
    {
      "element_id": "wdex-el-000001",
      "type": "paragraph",
      "text": "本文",
      "source_locator": {
        "part": "word/document.xml",
        "paragraph_index": 12,
        "page": 3
      },
      "children": []
    }
  ],
  "relations": [],
  "comments": [],
  "revisions": [],
  "media": []
}
```

## 5. OpenXML 解析要求

| ID | 要求 |
| --- | --- |
| WDEX-OXML-001 | `.docx` を ZIP パッケージとして読み取り、OpenXML パートを解析できること。 |
| WDEX-OXML-002 | `word/document.xml`、`word/styles.xml`、`word/numbering.xml`、`word/_rels/document.xml.rels` を解析できること。 |
| WDEX-OXML-003 | `docProps/core.xml`、`docProps/app.xml`、`docProps/custom.xml` から取得可能なメタデータを抽出できること。 |
| WDEX-OXML-004 | コメント、脚注、文末脚注、ヘッダー、フッター、画像、埋め込みオブジェクトに関する関連パートを解決できること。 |
| WDEX-OXML-005 | XML 名前空間、relationship target の相対パス、外部リンクを正規化して扱えること。 |
| WDEX-OXML-006 | パート欠落、不正 XML、未対応要素があっても、文書全体を即時破棄せず、可能な範囲の抽出結果と警告を返すこと。 |

## 18. 初期スコープ

初期実装では以下を優先する。

| 優先度 | 対象 |
| --- | --- |
| Must | `.docx` 取込、原本ハッシュ、メタデータ、章節、段落、run 書式、単純表、結合表、画像抽出、コメント、脚注、Markdown/JSON 出力、ZIP パッケージ保存 |
| Should | 多階層リスト番号復元、変更履歴、ブックマーク、外部リンク、図表キャプション候補、ページ推定値 |
| Could | テキストボックス、図形内テキスト、SmartArt、相互参照フィールド、ヘッダー・フッター詳細 |
| Later | `.doc` 直接取込、Word 完全レンダリング、OLE 内容解析、レイアウト完全再現 |

## 19. 非採用方針

参考情報に含まれていた以下の前提は、本 D2D 要件としては採用しない。

| 項目 | 方針 |
| --- | --- |
| Flask Web アプリ | D2D は Electron / React / Worker 構成を基本とするため、Flask 固有の API 仕様は採用しない。 |
| 単体 Web UI の左右プレビュー | D2D の原本ビュー、抽出データビュー、レビュー UI に統合する。 |
| `static/extracted/<session_id>/` 固定保存 | D2D のワークスペース、ZIP パッケージ、ジョブ成果物管理に従う。 |
| 抽出器単体での意味分類確定 | 要求、機能、制約、IF などの設計意味分類は、中間データ処理、LLM 候補生成、人間レビューで確定する。 |
| Markdown を正本とする運用 | 正本は構造化 JSON / JSONL とし、Markdown はレビュー・LLM 入力・エクスポート用の派生物とする。 |
