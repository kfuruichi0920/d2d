# PowerPoint文書情報抽出プラグイン 詳細要求

## 1. 位置づけ

本ドキュメントは、D2D(設計情報デジタル化・トレーサビリティ支援ツール)における PowerPoint 文書情報抽出機能の詳細要求を定義する。

PowerPoint 文書抽出機能は、`docs/srs_main.md` の以下に対応する詳細要求である。

- `3.1 対象文書` の PowerPoint 文書
- `7.1 原本取込` の PowerPoint 取込要求
- `7.2 文書構造抽出` のスライド、図形、テキストボックス、表、図、原本位置抽出要求
- `7.3 抽出結果レビュー` の原本表示と抽出結果表示の並列確認要求
- `8. 中間データ処理要求` のチャンク化、図表説明、参照リンク生成要求
- `11.3 図・表編集` の図情報表示・編集、構造化表編集要求

本機能は単独のクライアントサイド Web IDE ではなく、D2D のプラグイン構成、ジョブ管理、ZIP パッケージ管理、SQLite 管理、LLM Gateway、レビュー UI と連携する PowerPoint 取込・抽出プラグインとして実装する。

## 2. 基本方針

| ID | 要求 |
| --- | --- |
| PDEX-001 | PowerPoint 抽出機能は `plugin.powerpoint-importer` または同等の PowerPoint 取込プラグインとして提供すること。 |
| PDEX-002 | 抽出処理は UI スレッドから分離し、D2D のジョブ管理機構から実行できること。 |
| PDEX-003 | 原本 PowerPoint ファイルは改変せず、原本ハッシュ、ファイル名、取込日時、抽出器バージョンを保持すること。 |
| PDEX-004 | 抽出結果は、原本忠実な抽出データと、レビュー・LLM 入力向けに整理した中間データを分離して保持すること。 |
| PDEX-005 | スライド、テキスト、図形、コネクタ、画像、表、グループ、ノート、座標、レイヤー順は必須抽出観点として扱うこと。 |
| PDEX-006 | スライド上の読書順、タイトル、本文、図解、注記の判定は候補として生成し、人間レビューで確定できること。 |
| PDEX-007 | LLM を利用する場合でも、PPTX 内部構造、スライド順、要素座標、画像抽出、relationship 解決はルールベースで実施すること。 |
| PDEX-008 | LLM 出力は候補として保持し、抽出データそのものを上書きしないこと。 |
| PDEX-009 | 抽出結果には、後続の設計モデル要素が根拠リンクとして参照できる安定 ID を付与すること。 |

## 3. 対象ファイル

| ID | 要求 |
| --- | --- |

## 4. 抽出データモデル

PowerPoint 抽出結果は、プレゼンテーション単位のメタデータ、スライド、スライド要素、ノート、メディア、参照関係、抽出ログに分けて保持する。

| ID | 要求 |
| --- | --- |
| PDEX-MODEL-001 | 抽出データは JSON / JSONL として保存できること。 |
| PDEX-MODEL-002 | 抽出データは ZIP パッケージ内に格納できること。 |
| PDEX-MODEL-003 | 抽出データの schema_version を保持すること。 |
| PDEX-MODEL-004 | プレゼンテーション、スライド、テキスト、図形、コネクタ、画像、表、表セル、グループ、ノート、コメントに安定 ID を付与すること。 |
| PDEX-MODEL-005 | 各要素は `source_presentation_id`、`source_path`、`source_hash`、`slide_number`、`slide_id`、`element_path`、`source_locator`、`extraction_status` を保持すること。 |
| PDEX-MODEL-006 | `source_locator` は、OOXML パート名、relationship ID、スライド番号、図形 ID、要素階層、座標、z-order など、原本位置を再確認するための情報を保持すること。 |
| PDEX-MODEL-007 | 抽出時に確定できない情報は空文字で埋めず、未取得、対象外、推定、抽出失敗を区別できる状態値で表すこと。 |
| PDEX-MODEL-008 | Markdown 出力は閲覧・LLM 入力用の派生成果物とし、正本はスライド中心の構造化 JSON / JSONL とすること。 |

抽出データの概念構造は以下を基本とする。

```json
{
  "presentation": {
    "presentation_id": "ppt-...",
    "file_name": "sample.pptx",
    "source_hash": "sha256:...",
    "slide_size": { "width_emu": 12192000, "height_emu": 6858000 },
    "metadata": {},
    "statistics": {}
  },
  "slides": [
    {
      "slide_id": "slide-001",
      "slide_number": 1,
      "title": "スライドタイトル",
      "elements": [
        {
          "element_id": "pdex-el-000001",
          "type": "text",
          "text": "本文",
          "rect": { "x": 0, "y": 0, "w": 0, "h": 0, "unit": "emu" },
          "source_locator": { "part": "ppt/slides/slide1.xml", "shape_id": "2" }
        }
      ],
      "notes": []
    }
  ]
}
```

## 5. OOXML 解析要求

| ID | 要求 |
| --- | --- |
| PDEX-OXML-001 | `.pptx` 系ファイルを ZIP パッケージとして読み取り、OOXML パートを解析できること。 |
| PDEX-OXML-002 | `ppt/presentation.xml`、`ppt/_rels/presentation.xml.rels`、`ppt/slides/slideN.xml`、`ppt/slides/_rels/slideN.xml.rels` を解析できること。 |
| PDEX-OXML-003 | `ppt/theme/themeN.xml`、`ppt/slideMasters/`、`ppt/slideLayouts/` を可能な範囲で解析し、テーマ色、レイアウト、継承情報を保持できること。 |
| PDEX-OXML-004 | `docProps/core.xml`、`docProps/app.xml`、`docProps/custom.xml` から取得可能なメタデータを抽出できること。 |
| PDEX-OXML-005 | スライド relationship から画像、動画、音声、notesSlide、comments、embedded object への参照を解決できること。 |
| PDEX-OXML-006 | XML 名前空間、relationship target の相対パス、外部リンクを正規化して扱えること。 |
| PDEX-OXML-007 | パート欠落、不正 XML、未対応要素があっても、プレゼンテーション全体を即時破棄せず、可能な範囲の抽出結果と警告を返すこと。 |

## 19. 初期スコープ

初期実装では以下を優先する。

| 優先度 | 対象 |
| --- | --- |
| Must | `.pptx` / `.pptm` 取込、原本ハッシュ、メタデータ、スライド一覧、スライドサイズ、テキスト、図形、画像、表、ノート、座標、Markdown/JSON 出力、ZIP パッケージ保存 |
| Should | グループ座標、コネクタ、テーマ色、読書順候補、タイトル候補、スライド概要画像、コメント、リンク、レビュー UI での除外設定 |
| Could | スライドマスター継承詳細、アニメーション、動画・音声メディア、図形の意味分類候補、Vision LLM 連携 |
| Later | `.ppt` 直接解析、PowerPoint 完全レンダリング、アニメーション再現、OLE 内容解析、SmartArt 詳細解析 |

## 20. 非採用方針

参考情報に含まれていた以下の前提は、本 D2D 要件としては採用しない。

| 項目 | 方針 |
| --- | --- |
| クライアントサイド完結 IDE | D2D は Electron / React / Worker 構成を基本とするため、ブラウザ内完結を固定要件にしない。機密性はローカル worker と外部送信制御で担保する。 |
