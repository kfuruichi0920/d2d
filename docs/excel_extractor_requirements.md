# Excel文書情報抽出プラグイン 詳細要求

## 1. 位置づけ

本ドキュメントは、D2D(設計情報デジタル化・トレーサビリティ支援ツール)における Excel 文書情報抽出機能の詳細要求を定義する。

Excel 文書抽出機能は、`docs/srs_main.md` の以下に対応する詳細要求である。

- `3.1 対象文書` の Excel 文書、CSV / TSV ファイル
- `7.1 原本取込` の Excel 取込要求
- `7.2 文書構造抽出` の Excel セル、結合セル、行列構造、表、図、原本位置抽出要求
- `7.3 抽出結果レビュー` の原本表示と抽出結果表示の並列確認要求
- `8. 中間データ処理要求` のチャンク化、図表説明、参照リンク生成要求
- `11.3 図・表編集` の構造化表編集、行・列・セル単位の根拠利用要求

本機能は単独の FastAPI Web アプリや単独の React/Vite IDE ではなく、D2D のプラグイン構成、ジョブ管理、ZIPアーカイブ管理、設計情報ストア管理、LLM Gateway、レビュー UI と連携する Excel 文書抽出プラグインとして実装する。

## 2. 要求粒度の扱い

本ドキュメントは、[文書抽出プラグイン 共通設計要求](extractor_common_design_requirements.md) を前提とする形式別要求である。

ここに記載する XML パート名、内部ファイル名、ライブラリ名、UI 構成、処理アルゴリズムは、相互運用や検証のための参考情報であり、明示的に「必須」と書かれていない限り実装方式を固定しない。実装時は、共通データ契約、原本根拠の保持、レビュー可能性、機密情報保護、成果物再現性を満たす範囲で、より単純または堅牢な方式を選んでよい。

## 3. 基本方針

| ID | 要求 |
| --- | --- |
| XDEX-001 | Excel 抽出機能は `plugin.excel-importer` または同等の Excel 取込プラグインとして提供すること。 |
| XDEX-002 | 抽出処理は UI スレッドから分離し、D2D のジョブ管理機構から実行できること。 |
| XDEX-003 | 原本 Excel ファイルは改変せず、原本ハッシュ、ファイル名、取込日時、抽出器バージョンを保持すること。 |
| XDEX-004 | 抽出結果は、原本忠実な抽出データと、レビュー・LLM 入力向けに整理した中間データを分離して保持すること。 |
| XDEX-005 | Excel のシート、セル、結合セル、数式、コメント、名前付き範囲、非表示状態、表領域、ヘッダーは必須抽出観点として扱うこと。 |
| XDEX-006 | 表、文章、注記、画像、図形などの領域判定は候補として生成し、人間レビューで確定できること。 |
| XDEX-007 | LLM を利用する場合でも、ワークブック構造、セル座標、結合範囲、数式、スタイル、名前付き範囲、非表示状態の抽出はルールベースで実施すること。 |
| XDEX-008 | LLM 出力は候補として保持し、抽出データそのものを上書きしないこと。 |
| XDEX-009 | 抽出結果には、後続の設計モデル要素が根拠リンクとして参照できる安定 ID を付与すること。 |

## 5. 抽出データモデル

Excel 抽出結果は、ブック単位のメタデータ、シート、セル、結合セル、表領域、文章領域、画像、図形、コメント、名前付き範囲、抽出ログに分けて保持する。

| ID | 要求 |
| --- | --- |
| XDEX-MODEL-001 | 抽出データは JSON / JSONL として保存できること。 |
| XDEX-MODEL-002 | 抽出データは成果物フォルダに格納し、必要に応じてZIPアーカイブ化できること。 |
| XDEX-MODEL-003 | 抽出データの schema_version を保持すること。 |
| XDEX-MODEL-004 | ブック、シート、セル、結合セル、行、列、表、文章領域、画像、図形、コメント、名前付き範囲に安定 ID を付与すること。 |
| XDEX-MODEL-005 | 各要素は `source_workbook_id`、`source_path`、`source_hash`、`sheet_name`、`address`、`range`、`source_locator`、`extraction_status` を保持すること。 |
| XDEX-MODEL-006 | `source_locator` は、OOXML パート名、シート ID、シート名、セル番地、範囲、drawing relationship ID など、原本位置を再確認するための情報を保持すること。 |
| XDEX-MODEL-007 | 抽出時に確定できない情報は空文字で埋めず、未取得、対象外、推定、抽出失敗を区別できる状態値で表すこと。 |
| XDEX-MODEL-008 | Markdown 出力は閲覧・LLM 入力用の派生成果物とし、正本はシート中心の構造化 JSON / JSONL とすること。 |
| XDEX-MODEL-009 | 表抽出 JSON は `sheet`、`tables`、`table_id`、`range`、`title`、`columns`、`rows` を中心にした sheet-centric な構造を持つこと。 |

抽出データの概念構造は以下を基本とする。

```json
{
  "workbook": {
    "workbook_id": "wb-...",
    "file_name": "sample.xlsx",
    "source_hash": "sha256:...",
    "metadata": {},
    "statistics": {}
  },
  "sheets": [
    {
      "sheet_id": "sheet-001",
      "sheet": "Sheet1",
      "sheet_state": "visible",
      "used_range": "A1:H42",
      "cells": [],
      "merged_cells": [],
      "tables": [
        {
          "table_id": "table-001",
          "range": "A5:D10",
          "title": "表タイトル",
          "columns": ["項目", "内容"],
          "rows": []
        }
      ],
      "documents": [],
      "images": [],
      "shapes": [],
      "comments": [],
      "named_ranges": []
    }
  ]
}
```

## 6. OOXML 解析要求

| ID | 要求 |
| --- | --- |
| XDEX-OXML-001 | `.xlsx` 系ファイルを ZIP パッケージとして読み取り、OOXML パートを解析できること。 |
| XDEX-OXML-002 | `xl/workbook.xml`、`xl/_rels/workbook.xml.rels`、`xl/worksheets/sheetN.xml`、`xl/styles.xml`、`xl/sharedStrings.xml` を解析できること。 |
| XDEX-OXML-003 | `docProps/core.xml`、`docProps/app.xml`、`docProps/custom.xml` から取得可能なメタデータを抽出できること。 |
| XDEX-OXML-004 | シート relationship から drawings、comments、table definitions、media への参照を解決できること。 |
| XDEX-OXML-005 | XML 名前空間、relationship target の相対パス、外部リンクを正規化して扱えること。 |
| XDEX-OXML-006 | パート欠落、不正 XML、未対応要素があっても、ブック全体を即時破棄せず、可能な範囲の抽出結果と警告を返すこと。 |
| XDEX-OXML-007 | `openpyxl` 等のライブラリを利用できる場合でも、画像、図形、コメント、relationship 解決で不足する情報は OOXML 直解析で補完できること。 |

## 20. 参考実装から持ち込まない前提

参考情報に含まれていた以下の前提は、本 D2D 要件へそのまま持ち込まない。これは同等機能を別方式で実現することを妨げるものではなく、共通設計要求に従って実装時に再選択する。

| 項目 | 方針 |
| --- | --- |
| FastAPI 単体 API | D2D の共通アプリ基盤に統合するため、FastAPI 固有の API 仕様を要求として固定しない。 |
| React/Vite 単体 IDE | D2D の共通ビューへ統合する。ただし、必要な操作性を満たす専用ビューの追加は妨げない。 |
| `output/{session_id}` 固定保存 | D2D のワークスペース、成果物フォルダ、ZIPアーカイブ、ジョブ成果物管理に従う。 |
| 抽出器単体での意味分類確定 | 要求、機能、制約、IF、データモデルなどの設計意味分類は、中間データ処理、LLM 候補生成、人間レビューで確定する。 |
| Markdown を正本とする運用 | 正本は構造化 JSON / JSONL とし、Markdown はレビュー・LLM 入力・エクスポート用の派生物とする。 |
| LLM 前提の領域確定 | LLM は候補生成に限定し、ルールベース候補と人間レビューで完結できるようにする。 |
