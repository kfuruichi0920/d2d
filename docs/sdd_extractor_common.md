# 文書抽出機能 共通設計書

## 1. 目的

本ドキュメントは、Word、Excel、PowerPoint、PDF などの文書抽出機能に共通する設計を定義する。

各形式別の詳細設計書は、ここに示す共通設計を前提とし、形式固有の抽出観点だけを追加する。形式別文書に記載された XML パート名、ライブラリ名、処理アルゴリズム、画面構成は、明示的に「必須」と書かれていない限り、実装候補または検証観点であり、実装方式を固定するものではない。

## 2. 設計粒度の原則

| ID | 設計内容 |
| --- | --- |
| EXT-COM-001 | 設計書は、実装手順だけでなく、利用者価値、入出力契約、保持すべき根拠情報、レビュー可能性を中心に記述する。 |
| EXT-COM-002 | 形式別詳細設計書は、抽出できる情報、失敗時の扱い、レビュー時に必要な情報、後続モデルへの接続条件を定義すること。 |
| EXT-COM-003 | 具体的なライブラリ、フレームワーク、API エンドポイント、内部ファイル名、UI コンポーネント名は、原則として参考実装情報または実装候補として扱うこと。 |
| EXT-COM-004 | 必須設計観点は、原本を改変しないこと、根拠リンクを失わないこと、人間レビューで確定できること、外部送信を制御できること、再現可能な成果物を保存できることを中心に定義すること。 |
| EXT-COM-005 | 抽出精度や対応範囲は、設計ID、対応状態、制約、未対応理由を明示して管理できること。 |
| EXT-COM-006 | LLM が実装判断を行う余地を残すため、同じ成果を満たす複数の実装方式を許容すること。 |

## 3. 共通アーキテクチャ設計

| ID | 設計内容 |
| --- | --- |
| EXT-COM-010 | 各抽出器は D2D の文書抽出機能に含まれる形式別機能として扱い、有効化、無効化、有効化条件、設定を管理できること。 |
| EXT-COM-011 | 重い取込、抽出、レンダリング、OCR、LLM 処理はジョブとして実行し、UI と分離すること。 |
| EXT-COM-012 | 実装は Electron / React / Worker 構成を基本とするが、個別処理は Node.js、Python、Rust、Go、外部 CLI 等から適切な手段を選択できること。 |
| EXT-COM-013 | renderer は原則として直接ファイル I/O や機密情報管理を行わず、アプリ基盤の安全な API 境界を通じて処理すること。 |
| EXT-COM-014 | 形式別抽出器は、共通のジョブ状態、ログ、成果物管理、レビュー状態、エラー通知を利用できること。 |
| EXT-COM-015 | 抽出器ごとに完全な UI を重複実装せず、原本ビュー、抽出データビュー、レビュー UI、ログビュー、成果物ビューを共通化できる設計とすること。 |

## 4. 共通データ設計

| ID | 設計内容 |
| --- | --- |
| EXT-COM-020 | 抽出成果物は、原本データ、抽出データ、中間データ、レビュー確定データを区別して保持できること。 |
| EXT-COM-021 | 抽出データは JSON / JSONL を基本とし、schema_version、抽出器名、抽出器バージョン、抽出日時、抽出設定を保持すること。 |
| EXT-COM-022 | 各抽出要素は、安定 ID、要素種別、原本識別子、原本位置、抽出方式、抽出状態、警告、レビュー状態を保持できること。 |
| EXT-COM-023 | 原本位置は、形式に応じてページ番号、シート名、セル範囲、スライド番号、座標、段落位置、XML パート等を組み合わせて表現できること。 |
| EXT-COM-024 | 原本位置の内部表現は形式ごとに異なってよいが、設計モデルの source_links から再確認できること。 |
| EXT-COM-025 | 抽出できない情報は空文字や推測値で埋めず、未取得、対象外、推定、抽出失敗、未対応を区別できること。 |
| EXT-COM-026 | Markdown、HTML、画像、ページ画像・スライド画像等の確認用レンダリング成果物は派生成果物として扱い、正本は構造化データとレビュー確定データとすること。 |
| EXT-COM-027 | ZIPアーカイブには manifest を含め、原本ハッシュ、成果物一覧、schema_version、抽出器バージョンを保持すること。 |
| EXT-COM-028 | ZIPアーカイブは成果物コンテナであり、内部に JSON / JSONL、SQLite、画像、ページ画像・スライド画像等の確認用レンダリング成果物、ログ等を含める場合は、それぞれの役割を manifest で識別できること。 |
| EXT-COM-029 | SQLite は抽出成果物またはレビュー確定データの保存形態として利用できるが、抽出データの唯一の正本形式として固定しないこと。 |

## 5. 抽出と候補生成の分担

| ID | 設計内容 |
| --- | --- |
| EXT-COM-030 | 構造の取得、位置の保持、ID 付与、成果物保存、根拠リンク作成は決定的処理を優先すること。 |
| EXT-COM-031 | 見出し判定、表領域判定、読み順、図表説明、要求候補、関係候補など意味判断を含む処理は候補として扱うこと。 |
| EXT-COM-032 | LLM は候補生成、補正、要約、説明生成、レビュー補助に利用できるが、確定情報への反映には人間レビューを必要とすること。 |
| EXT-COM-033 | LLM へ渡す入力は、原本根拠リンク、抽出要素 ID、必要な範囲の本文または画像に限定できること。 |
| EXT-COM-034 | 外部 LLM API が無効なワークスペースでは、ローカル処理と手動レビューで作業継続できること。 |
| EXT-COM-035 | LLM 候補には、入力、プロンプト、モデル、実行日時、根拠、確度、採否状態を紐づけられること。 |

## 6. レビュー設計

| ID | 設計内容 |
| --- | --- |
| EXT-COM-040 | 抽出結果は未確認、確認済、要修正、棄却、採用済などの状態で管理できること。 |
| EXT-COM-041 | 人間は抽出要素の種別、範囲、本文、表データ、キャプション、読み順、根拠リンクを修正できること。 |
| EXT-COM-042 | レビュー操作は履歴として保持し、抽出時点の値とレビュー確定値を区別できること。 |
| EXT-COM-043 | 原本ビューと抽出データビューを並べて、抽出要素から原本該当箇所を確認できること。 |
| EXT-COM-044 | 警告、未対応要素、推定値、LLM 候補はレビュー対象として一覧化できること。 |

## 7. 出力・連携設計

| ID | 設計内容 |
| --- | --- |
| EXT-COM-050 | LLM 入力向け Markdown は、根拠 ID を保持しつつ、ページ番号、装飾、内部メタデータなど不要情報を除外できること。 |
| EXT-COM-051 | レビュー向け Markdown は、抽出要素 ID、原本位置、警告、レビュー状態を表示できること。 |
| EXT-COM-052 | 表データは必要に応じて JSON 配列、Markdown 表、SQLite テーブル、CSV / TSV に出力できること。 |
| EXT-COM-053 | 図・画像・ページレンダリング・スライドレンダリング等の画像成果物は、相対パスとハッシュで管理できること。 |
| EXT-COM-054 | 抽出済み要素は、中間データ処理、設計モデル、トレース、図表説明、関係候補生成から参照できること。 |

## 8. 実装裁量として残す事項

以下は設計として固定しない。実装時に技術検証、性能、ライセンス、保守性を踏まえて選択する。

- 使用ライブラリ、外部 CLI、ワーカー言語
- 内部 API エンドポイント設計
- 一時ディレクトリ名、セッション ID 形式、内部ファイル名
- XML / PDF 内部パートの読み取り順序
- UI コンポーネント分割、状態管理ライブラリ、描画方式
- OCR エンジン、表抽出エンジン、レンダリングエンジン
- LLM プロバイダ、モデル、プロンプト詳細

## 9. 形式別設計書の読み方

形式別設計書の各 ID は、その形式で失ってはいけない情報とレビュー観点を示す。実装者は、共通設計と形式別設計を満たす限り、より単純な方式、より堅牢な方式を選んでよい。

特に、`OOXML 解析設計`、`PDF 解析・レンダリング設計`、`出力設計` に含まれる具体名は、相互運用や検証のための目安であり、利用者に見える機能やデータ契約を満たすことを優先する。
