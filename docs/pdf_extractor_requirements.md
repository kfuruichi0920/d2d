# PDF文書情報抽出プラグイン 詳細要求

## 1. 位置づけ

本ドキュメントは、D2D(設計情報デジタル化・トレーサビリティ支援ツール)における PDF 文書情報抽出機能の詳細要求を定義する。

PDF 文書抽出機能は、`docs/srs_main.md` の以下に対応する詳細要求である。

- `3.1 対象文書` の PDF 文書
- `7.1 原本取込` の PDF 取込要求
- `7.2 文書構造抽出` の PDF テキスト、表、図、ページ位置抽出要求
- `7.3 抽出結果レビュー` の原本表示と抽出結果表示の並列確認要求
- `8. 中間データ処理要求` のチャンク化、図表説明、参照リンク生成要求
- `11.3 図・表編集` の図情報表示・編集、構造化表編集要求

本機能は単独の FastAPI + React/Vite IDE ではなく、D2D のプラグイン構成、ジョブ管理、ZIPアーカイブ管理、設計情報ストア管理、LLM Gateway、レビュー UI と連携する PDF 文書抽出プラグインとして実装する。

## 2. 要求粒度の扱い

本ドキュメントは、[文書抽出プラグイン 共通設計要求](extractor_common_design_requirements.md) を前提とする形式別要求である。

ここに記載する XML パート名、内部ファイル名、ライブラリ名、UI 構成、処理アルゴリズムは、相互運用や検証のための参考情報であり、明示的に「必須」と書かれていない限り実装方式を固定しない。実装時は、共通データ契約、原本根拠の保持、レビュー可能性、機密情報保護、成果物再現性を満たす範囲で、より単純または堅牢な方式を選んでよい。

## 3. 基本方針

| ID | 要求 |
| --- | --- |
| PDFX-001 | PDF 抽出機能は `plugin.pdf-importer` または同等の PDF 取込プラグインとして提供すること。 |
| PDFX-002 | 抽出処理、ページ画像レンダリング、OCR、LLM 補助処理は UI スレッドから分離し、D2D のジョブ管理機構から実行できること。 |
| PDFX-003 | 原本 PDF ファイルは改変せず、原本ハッシュ、ファイル名、取込日時、抽出器バージョンを保持すること。 |
| PDFX-004 | 抽出結果は、原本忠実な抽出データと、レビュー・LLM 入力向けに整理した中間データを分離して保持すること。 |
| PDFX-005 | ページ、テキストブロック、表、画像、数式、注釈、リンク、ページ座標、レンダリング画像は必須抽出観点として扱うこと。 |
| PDFX-006 | PDF の読み順、段落、見出し、表題、キャプション、数式種別、図表説明は候補として生成し、人間レビューで確定できること。 |
| PDFX-007 | LLM を利用する場合でも、PDF メタデータ、ページ寸法、抽出座標、ブロック ID、画像クロップ、成果物保存はルールベースで実施すること。 |
| PDFX-008 | LLM 出力は候補として保持し、抽出データそのものを上書きしないこと。 |
| PDFX-009 | 抽出結果には、後続の設計モデル要素が根拠リンクとして参照できる安定 ID を付与すること。 |

## 5. 抽出データモデル

PDF 抽出結果は、文書単位のメタデータ、ページ、ブロック、表、画像、数式、注釈、リンク、レンダリング画像、抽出ログに分けて保持する。

| ID | 要求 |
| --- | --- |
| PDFX-MODEL-001 | 抽出データは JSON / JSONL として保存できること。 |
| PDFX-MODEL-002 | 抽出データは成果物フォルダに格納し、必要に応じてZIPアーカイブ化できること。 |
| PDFX-MODEL-003 | 抽出データの schema_version を保持すること。 |
| PDFX-MODEL-004 | 文書、ページ、テキストブロック、見出し候補、表、表セル、画像、数式、注釈、リンク、クロップ画像に安定 ID を付与すること。 |
| PDFX-MODEL-005 | 各要素は `source_pdf_id`、`source_path`、`source_hash`、`page_number`、`bbox`、`source_locator`、`extraction_status` を保持すること。 |
| PDFX-MODEL-006 | `bbox` は PDF ページ座標を正本として保持し、必要に応じてレンダリング画像座標へ変換できること。 |
| PDFX-MODEL-007 | `source_locator` は、ページ番号、座標、抽出方式、クロップ画像 ID、テキスト抽出エンジン、OCR 実行有無など、原本位置を再確認するための情報を保持すること。 |
| PDFX-MODEL-008 | 抽出時に確定できない情報は空文字で埋めず、未取得、対象外、推定、抽出失敗を区別できる状態値で表すこと。 |
| PDFX-MODEL-009 | Markdown 出力は閲覧・LLM 入力用の派生成果物とし、正本はページ中心の構造化 JSON / JSONL とすること。 |

抽出データの概念構造は以下を基本とする。

```json
{
  "document": {
    "pdf_id": "pdf-...",
    "file_name": "sample.pdf",
    "source_hash": "sha256:...",
    "metadata": {},
    "statistics": {}
  },
  "pages": [
    {
      "page_id": "page-001",
      "page_number": 1,
      "width": 595.27,
      "height": 841.89,
      "unit": "pt",
      "blocks": [
        {
          "block_id": "pdfx-block-000001",
          "type": "text",
          "bbox": [50.0, 100.0, 545.0, 150.0],
          "text": "本文",
          "source_locator": { "page_number": 1, "method": "text_layer" }
        }
      ]
    }
  ]
}
```

## 18. 初期スコープ

初期実装では以下を優先する。

| 優先度 | 対象 |
| --- | --- |
| Must | PDF 取込、原本ハッシュ、メタデータ、ページ一覧、ページ寸法、ページ画像、テキストブロック、表候補、画像候補、bbox、Markdown/JSON 出力、成果物フォルダ保存、必要に応じたZIPアーカイブ化 |
| Should | OCR 候補、クロップ画像、表 SQLite 出力、読書順候補、見出し/キャプション候補、注釈、リンク、レビュー UI でのブロック編集 |

| Later | PDF 完全レイアウト再現、フォーム入力値詳細、署名検証、埋め込みファイル詳細解析、高度な数式認識 |

## 19. 参考実装から持ち込まない前提

参考情報に含まれていた以下の前提は、本 D2D 要件へそのまま持ち込まない。これは同等機能を別方式で実現することを妨げるものではなく、共通設計要求に従って実装時に再選択する。

| 項目 | 方針 |
| --- | --- |
| FastAPI 単体 API | D2D の共通アプリ基盤に統合するため、FastAPI 固有の API 仕様を要求として固定しない。 |
