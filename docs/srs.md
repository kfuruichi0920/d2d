## 1. 目的

本ツールは、既存のWord、Excel、PowerPoint、Visio、PDF、テキスト等で作成された自然言語中心の設計文書を段階的にデジタル化し、設計要素および設計要素間の関係性を構造化管理することを目的とする。

これにより、要求、制約、機能、構造、振舞、状態、インタフェース、データモデル、検証情報、設計判断根拠等の間に複数観点のトレーサビリティを確保し、仕様変更時の影響分析、設計根拠確認、検証カバレッジ確認、設計レビュー支援を機械的に実施可能にする。

本ツールは、すべてをLLMに自動判断させるものではなく、LLMを候補生成・要約・抽出・レビュー補助に用い、人間による逐次確認、修正、採用、棄却を前提とした human-in-the-loop 型の設計支援ツールとする。

## 2. 基本方針

### 2.1 データ階層

本ツールは、以下の4階層でデータを管理する。

| 階層 | 名称    | 内容                       | 主な役割            |
| -- | ----- | ------------------------ | --------------- |
| ①  | 原本データ | Word、Excel、PDF等の既存文書     | 改変しない一次情報(既存の仕様書・設計書)       |
| ②  | 抽出データ | タイトル、本文、図、表、数式等     | ①から原本ファイル単位で抽出した外形情報を保持      |
| ③  | 中間データ | 統合設計書、原本に近い自然言語本文、章構成、図表説明、参照関係、段落関係、SQLite DB等     | 複数ファイルに分かれた設計書を成果物単位で統合し、原本に近い文書出力と設計情報整理の両方に利用する |
| ④  | 設計モデル | 要求、制約、機能、構造、振舞、状態、IF、検証、要素間関係等 | ③の設計結果を根拠に、階層構造と粒度を持つ要素・関係性モデルで表現した形式    |

**データ変換フロー**

```mermaid

    B["②抽出データ\n(ファイル単位の章節・段落・図表・数式)"]
    C["③中間データ\n(統合設計書・自然言語本文・章構成)"]
    D["④設計モデル\n(階層化された要求・機能・構造・関係性)"]

    B -.->|"DB to Text\n(差分表示・LLM入力用)"| T[(Text化)]
    C -.->|"DB to Text\n(差分表示・LLM入力用)"| T
```

### 2.2 データ管理原則

1. ①原本データは改変しない。
2. ②抽出データは①から抽出した情報であり、原則として原本ファイル単位で管理する。
3. ②抽出データは原本忠実性を重視し、意味解釈を過度に入れない。
4. ③中間データは、設計書が複数ファイルに分かれている場合でも、開発プロセス上の成果物単位で統合し、原本に近い自然言語本文と章構成を保持した統合設計書データとして扱う。
5. ③中間データは、章構成、原本設計書のアウトライン（親子関係）、本文、図表、参照関係を持ったデータ化された設計書であり、レポート機能により原本に近い文書として出力できる。②と④とのトレーサビリティ、③の各成果物間のトレーサビリティを持つ。
6. ④設計モデルで設計上の意味を確定する。
7. ④設計モデルは、要素と関係性を階層構造および粒度を考慮して接続し、③の設計結果との対応関係を保持する。
8. IDをもつ設計情報や各要素・関係に対して双方向トレーサビリティを確保する。
9. LLMの出力は確定情報ではなく、候補情報として扱う。
10. 人間レビューにより採用・修正・棄却された情報のみを②抽出データ、③中間データ、④設計モデルの各正本へ反映する。

### 2.3 設計モデル化の工程

本ツールは、原本文書を直接④設計モデルへ変換するのではなく、以下の工程に分けて段階的に構造化・意味化し、トレーサブルな設計知識グラフとして統合する。

| 工程 | 位置づけ | 主な処理 |
| --- | --- | --- |
| ①原本データ→②抽出データ | 文書構造の取得 | 章、節、段落、表、図、箇条書き、セル、キャプション、参照等を文書階層として取得し、出典位置を保持する |
| ②抽出データ→③中間データ | 設計記述要素の抽出 | 文書中の記述単位を抽出し、要求らしさ、制約らしさ、IFらしさ等の候補属性、根拠位置、信頼度を保持する |
| ②抽出データ→③中間データ | 記述型への分類 | 段落、表、図、シナリオ、状態遷移表、IF一覧表、データ項目表、モデル図等の記述型へ分類する |
| ③中間データ→④設計モデル | 設計意味候補への昇格 | 記述資源を要求候補、機能候補、IF候補、状態候補、データ候補、制約候補、根拠候補等へ変換する |
| ③中間データ→④設計モデル | 同一対象の統合・正規化 | 重複、類似、別表現、別成果物上の同一対象を統合候補として扱い、UID、名称、型、粒度、属性、表記揺れ、単位、関係名候補を標準化する |
| ③中間データ→④設計モデル | 関係付与 | 設計トレース、分解、実現、割当、依存、検証、根拠等の関係を付与する |
| 双方向トレーサビリティ分析 | 検査 | 未接続、不整合、根拠不足、粒度不一致、循環、過剰な汎用関係利用等を検出する |

各工程の出力は候補を含む。特に設計意味候補への昇格、同一対象の統合・正規化、関係付与の結果は、人間レビューにより採用・修正・棄却されるまで④設計モデルの確定情報として扱わない。検査は新たなデータ階層を生成する工程ではなく、双方向トレーサビリティの分析機能により、①〜④および設計関係層を横断して実現する。

### 2.4 記述形式と設計意味の分離

設計書上の記述形式と設計上の意味は一致しない。例えば表は、インターフェース定義、状態遷移表、データ定義、試験条件、要求一覧、制約一覧のいずれにもなり得る。同様に、段落、図、箇条書き、シナリオ、モデル記述も文脈により異なる設計意味を持つ。

本ツールは、記述形式を先に安定して抽出・分類し、その後で設計意味候補へ昇格する。1つの記述資源が複数の設計意味候補を持つことを許容し、根拠、信頼度、LLM実行参照、レビュー状態を保持したうえで、人間レビューにより確定する。

同一対象の統合・正規化では、ノードの同一性、名称、型、粒度、属性を整える。関係付与は、ノード間の設計上の意味関係を付与する処理であり、統合・正規化とは独立してレビュー、修正、再生成できること。関係の妥当性、未接続、不整合、循環等の検査は、双方向トレーサビリティ分析機能で実現すること。

## 3. 対象範囲

### 3.1 対象文書

本ツールは、以下の原本文書を扱えること。

| 種別 |
| --- |
| Word文書(*2) |
| Excel文書(*2) |
| PowerPoint文書(*2) |
| Visio文書(*2) |
| PDF文書 |
| テキストファイル |
| Markdownファイル |
| CSV / TSVファイル |
| JSON / JSONL / YAMLファイル |
| ZIPアーカイブ(*1) |

レビュー記録、障害管理台帳、議事録、QA表、変更要求一覧など、設計・試験・レビュー活動で発生する管理文書も対象文書として扱う。

(*1)本ツールは、抽出データおよび中間データの成果物セットをZIP圧縮したアーカイブとしても扱えること。通常の作業成果物は、単一の SQLite DB（project.db）とDB外ファイル（blobs/）で一体管理し、ZIP圧縮前のプロジェクトルートをGit履歴管理対象にできること。manifestはZIPアーカイブ生成時のみ作成する。ZIP圧縮ファイルは、過去時点のアーカイブ、受け渡し、最新成果物との差分比較用インポートに用いる。

(*2)Word、Excel、PowerPoint、Visio文書はOpenXML系形式（.docx / .xlsx / .pptx / .vsdx）を対象とする。旧バイナリ形式（.doc / .xls / .ppt / .vsd）は対象外とし、必要な場合は事前にOpenXML系形式へ変換してから取り込む。

```mermaid
graph TD
```

## 6. データ管理要求

②抽出データ、③中間データ、④設計モデルの正本情報は、単一の SQLite DB（project.db）とDB外ファイル（blobs/）で管理する。JSON / JSONL、DB to Text、SQLite dump、差分表示用テキストは、②③④に共通する派生成果物または一時成果物として扱う。

### 6.1 成果物セット / ZIPアーカイブ管理

## 9. 設計モデル要求

### 9.1 設計要素管理

④設計モデルは、設計要素と設計要素間の関係を分離して管理する。設計要素の分類は以下の13分類とし、各設計要素は自らの分類を属性として保持できること。設計要素は `entity_registry` に共通登録し、種別固有の情報は `resource_*` 詳細テーブルで管理する。

| 分類 | 略号 | 対象要素 | 主な格納先 |
| --- | --- | --- | --- |
| 一次情報 | SRC | 原本文書、章、節、段落、図、表、数式、注記 | `source_document`、`source_location`、`resource_text/table/figure/formula` |
| 規範情報 | STD | 法律、規則、規約、業務規則、社内標準、外部標準 | `resource_reference`、`resource_text`、`resource_metadata` |
| 要求情報 | REQ | 上位要求、派生要求、機能要求、非機能要求 | `resource_text`、`resource_table`、`resource_metadata` |
| 制約情報 | CST | 法規制約、性能制約、安全制約、運用制約、実装制約 | `resource_text`、`resource_formula`、`resource_metadata` |
| 機能情報 | FUNC | 機能、サブ機能、機能責務 | `resource_text`、`resource_scenario`、`resource_model` |
| 構造情報 | STRUCT | 装置、ソフトウェア、モジュール、コンポーネント、タスク | `resource_model`、`resource_code`、`resource_metadata` |
| 振舞情報 | BEH | シナリオ、処理手順、イベント、アクション | `resource_scenario`、`resource_model` |

### 9.2 関係性管理

`trace_link.relation_type` は、関係種別の増加による意味重複、入力判断の揺れ、検索・分析時の複雑化を避けるため、以下の11種類に限定する（詳細DDLは `sdd_data_structure.md` 参照）。

| 関係種別 | 分類 | 用途 | 推奨方向 |
| --- | --- | --- | --- |
| `based_on` | 根拠関係 | 設計要素または関係が、原本、抽出データ、中間データ、判断根拠に基づくことを表す | 設計要素・関係 → 根拠 |
| `satisfies` | 設計意味関係 | 機能・構造・振舞・状態・IF・データ・実装等が、規範・要求・制約を満たすことを表す | 設計要素 → STD/REQ/CST |
| `allocated_to` | 設計意味関係 | 要求・制約・機能・責務を、それを担う設計要素へ割り当てることを表す | REQ/CST/FUNC/BEH → 担当設計要素 |
| `verifies` | 実装・検証関係 | 試験、レビュー、解析、検証情報が、要求・制約・設計要素・実装を検証することを表す | VERIF → 検証対象 |
| `contains` | 設計意味関係 | 構造要素、IF、データ構造等が下位要素を構造的に包含することを表す | 上位構造 → 下位構造 |
| `decomposes` | 設計意味関係 | 上位要素を下位・詳細・具体要素へ展開することを表す | 上位要素 → 下位要素 |
| `implements` | 実装・検証関係 | ソースコード、設定、DDL、モデル等の実装要素が設計要素を実装することを表す | IMPL → 設計要素 |
| `uses` | 設計意味関係 | 設計要素が、IF、データ、状態、外部資源等を利用することを表す | 利用元 → 利用対象 |
| `calls` | 実装・検証関係 | 実装要素、API、処理、振舞間の呼び出し関係を表す | 呼び出し元 → 呼び出し先 |
| `conflicts_with` | 設計意味関係 | 要求・制約・設計判断・設計要素間の競合または矛盾可能性を表す | 競合元 → 競合先 |
| `relates_to` | 暫定・補助 | 分類未確定の暫定的・補助的な関連を表す | 関連元 → 関連先 |

`derived_from`、`normalized_from`、`owns`、`realizes`、`depends_on`、`inputs`、`outputs`、`constrains`、`impacts`、`refines` は relation_type として採用しない。代替表現は以下とする。

| 非採用 relation_type | 代替表現 |
| --- | --- |
| `derived_from` | `based_on + basis_kind` |
| `normalized_from` | `based_on + basis_kind=normalized` |
| `owns` | `owner_uid` |
| `realizes` | `satisfies` または `implements` |
| `depends_on` | `uses`、`calls`、または関係探索により導出 |
| `inputs` | `uses + usage_kind=input` |
| `outputs` | `uses + usage_kind=output` |
| `constrains` | 設計要素が制約を `satisfies` する関係で表現 |
| `impacts` | 関係探索により導出 |
| `refines` | `decomposes + decomposition_kind=refinement` |

**注意**: 文書構成上の親子関係（章節配下等）は `trace_link` ではなく、`extracted_document.structure_json` および `intermediate_document.structure_json` 内のJSON構造として管理する。設計意味上の構造包含は `contains`、意味分解・詳細化は `decomposes`、入力・出力は `uses + usage_kind` で表現する。

### 9.3 関係属性

`trace_link` は、関係種別だけで意味を増やさず、必要な差分を属性で表現する。

| 対象 | 属性 | 値・用途 |
| --- | --- | --- |
| 共通 | `confidence` | 自動抽出・推論結果の信頼度 |
| 共通 | `created_by` | `human / rule / llm` |
| 共通 | `review_status` | `draft / review / approved / rejected / provisional` |
| 共通 | `rationale` | 関係設定の理由・根拠説明 |
| `based_on` | `basis_kind` | `original / extracted / normalized / inferred / human_approved` |
| `based_on` | `evidence_span` | 根拠範囲、ページ、セル範囲、段落範囲等 |
| `based_on` | `transform_note` | 抽出・正規化・判断の補足 |
| `allocated_to` | `allocation_kind` | `structure / behavior / state / interface / data` |
| `allocated_to` | `allocation_role` | `primary / supporting` |
| `decomposes` | `decomposition_kind` | `structural / functional / behavioral / logical / refinement` |
| `uses` | `usage_kind` | `input / output / read / write / update / publish / subscribe` |

### 9.4 設計要素間の許容関係

主要な許容関係は以下とする。詳細な実装ルールは `sdd_data_structure.md` の `relation_rule_master` に定義する。

| relation_type | 主な許容 Source | 主な許容 Target | ルール |
| --- | --- | --- | --- |
| `based_on` | STD/REQ/CST/FUNC/STRUCT/BEH/STATE/IF/DATA/VERIF/MGMT/IMPL/trace_link | SRC/中間要素/判断根拠 | 根拠関係専用。SRCは設計意味関係マトリクスから除外する |
| `satisfies` | FUNC/STRUCT/BEH/STATE/IF/DATA/IMPL | STD/REQ/CST | 要求・制約・規範への充足関係。要求側からは逆引き表示する |
| `allocated_to` | REQ/CST/FUNC/BEH | STRUCT/BEH/STATE/IF/DATA | 要求・制約・機能・責務の割当。FUNCを中心に設計要素を束ねる |
| `contains` | STRUCT/DATA/IF | STRUCT/DATA/IF | 構造的包含に限定する |
| `decomposes` | STD/REQ/CST/FUNC/BEH/STATE/IF/VERIF/MGMT | 同種または詳細要素 | 詳細化は `decomposition_kind=refinement` とする |
| `implements` | IMPL | FUNC/STRUCT/BEH/STATE/IF/DATA | 実装から設計要素への対応。IMPL→REQは直接保持せず、設計要素経由で辿る |
| `uses` | STRUCT/BEH/STATE/IF | STATE/IF/DATA | 入出力・読書き・発行購読は `usage_kind` で表す |
| `calls` | IMPL/IF/BEH | IMPL/IF/BEH | 呼び出し関係が明確な場合に限定する |
| `verifies` | VERIF | STD/REQ/CST/FUNC/STRUCT/BEH/STATE/IF/DATA/IMPL | 検証情報から検証対象へ向ける |
| `conflicts_with` | 任意の設計要素、MGMT、IMPL | 任意の設計要素、MGMT、IMPL | 文脈依存の競合として属性を必ず確認する |
| `relates_to` | 任意 | 任意 | 暫定リンク専用。レビュー後は可能な限り他の relation_type へ置換する |
